# SQLite Transactions: Source-Code Instrumentation Report

**Course:** Big Data Engineering  
**Topic:** SQLite transaction internals through source-code experiments

## System Overview

SQLite is an embedded relational database engine. Its transaction system must preserve ACID behavior while storing the whole database in ordinary files and coordinating access through the operating system VFS.

This submission studies transactions by modifying SQLite source code under `sqlite-master/src/`, rebuilding SQLite, and running experiments against the rebuilt binary. The experiments do not use mock traces.

## Transaction Execution Path

A write transaction follows this path:

`SQL statement -> sqlite3_exec/sqlite3_step -> VDBE bytecode -> B-tree insert -> pager begin -> lock acquisition -> rollback journal or WAL write -> commit phase one -> commit phase two`

Relevant code references:

- `sqlite-master/src/vdbe.c`: executes bytecode generated for SQL statements.
- `sqlite-master/src/btree.c:sqlite3BtreeInsert()`: inserts or overwrites B-tree cells for table/index writes.
- `sqlite-master/src/pager.c:sqlite3PagerBegin()`: begins a write transaction at the pager layer.
- `sqlite-master/src/pager.c:pagerLockDb()`: requests SQLite file-lock transitions.
- `sqlite-master/src/pager.c:sqlite3PagerCommitPhaseOne()`: syncs journal/WAL and writes dirty pages as needed.
- `sqlite-master/src/pager.c:sqlite3PagerCommitPhaseTwo()`: finalizes the transaction and releases/adjusts pager state.
- `sqlite-master/src/wal.c:sqlite3WalBeginWriteTransaction()` and `walFrames()`: handle WAL write transactions and frame commits.

## Design Decisions

### 1. Rollback Journal Atomic Commit

In rollback-journal mode, SQLite records old page images before committing changes. If a crash leaves a hot journal, the pager can replay it and restore the pre-transaction database state. The relevant paths are in `pager.c`, especially `hasHotJournal()`, `pager_playback()`, `sqlite3PagerCommitPhaseOne()`, and `sqlite3PagerCommitPhaseTwo()`.

### 2. WAL Append-Based Commit

In WAL mode, SQLite appends changed page frames to the WAL file instead of immediately overwriting the database file. This allows readers to continue using the database snapshot while one writer appends to WAL. The key paths are `wal.c:sqlite3WalBeginWriteTransaction()`, `wal.c:sqlite3WalFrames()`, and `wal.c:walFrames()`.

### 3. Pager Abstraction, Page Cache, and Lock Management

The B-tree layer does not manage filesystem locking or crash recovery directly. It asks the pager for writable pages. The pager coordinates page cache state, journal mode, lock level, sync behavior, and recovery. This keeps storage logic centralized but makes `pager.c` a critical subsystem.

## Concept Mapping

| Concept | SQLite implementation | Source reference |
|---|---|---|
| B-tree storage | Tables and indexes are stored as B-tree pages; inserts modify B-tree cells. | `sqlite-master/src/btree.c` |
| Write-ahead logging / rollback journaling | WAL appends frames; rollback journal stores old page images for undo. | `sqlite-master/src/wal.c`, `sqlite-master/src/pager.c` |
| Fault tolerance / recovery | Hot journal detection and playback recover from interrupted transactions. | `pager.c:hasHotJournal()`, `pager_playback()` |
| Locking / concurrency control | Pager lock transitions enforce SQLite's single-writer model. | `pager.c:pagerLockDb()` |

## Failure Analysis

### Data Size Increase

As data size grows, B-tree pages split and the tree height can increase. More pages must be read and written, and page-cache pressure rises. The relevant implementation areas are `btree.c` for page balancing/splitting and pager/pcache code for page caching. Larger transactions can still reduce commit overhead, but they do not remove B-tree growth costs.

### Crash During Transaction

In rollback-journal mode, an interrupted transaction should leave either no committed change or a journal that can be used for recovery. The hot-journal path in `pager.c` exists to detect and roll back incomplete transactions. The crash experiment verifies this behavior using the modified binary rather than assuming it.

### Concurrent Writers / Lock Contention

SQLite permits only one writer at a time. A competing writer may receive `SQLITE_BUSY` when it cannot obtain the required lock. This behavior maps to pager lock acquisition in `pager.c:pagerLockDb()` and lower VFS locking code. WAL improves reader/writer overlap, but it does not allow multiple simultaneous writers.
