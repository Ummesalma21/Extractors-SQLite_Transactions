SQLite is an embedded relational database engine. Its transaction system has to make ordinary file writes behave like ACID database transactions: changes must be atomic, durable after commit, isolated from conflicting writers, and recoverable after a crash.

This project studies that transaction system by inserting `[SQLITE_TRACE]` logs directly into SQLite source code, rebuilding SQLite, and running the modified binary. The relevant modified files are:

- `sqlite-master/src/vdbe.c`
- `sqlite-master/src/pager.c`
- `sqlite-master/src/btree.c`
- `sqlite-master/src/wal.c`

The source trace is used to understand one concrete execution path through SQLite internals. The behavior experiments then stress or compare transaction behavior using the same custom SQLite binary.

## 2. Execution Understanding / Complete Transaction Path

A write transaction follows this path:

`SQL statement -> sqlite3_step/sqlite3_exec -> VDBE bytecode -> transaction opcode -> pager begin -> lock acquisition -> B-tree insert -> journal/WAL work -> commit phase one -> commit phase two`

Important code locations:

- `sqlite-master/src/vdbe.c`: executes VDBE bytecode and transaction opcodes.
- `sqlite-master/src/btree.c:sqlite3BtreeInsert()`: inserts or overwrites B-tree cells.
- `sqlite-master/src/pager.c:sqlite3PagerBegin()`: starts the pager-level write transaction.
- `sqlite-master/src/pager.c:pagerLockDb()`: requests lock transitions such as shared, reserved, and exclusive locks.
- `sqlite-master/src/pager.c:hasHotJournal()`: checks whether rollback-journal recovery is needed.
- `sqlite-master/src/pager.c:sqlite3PagerCommitPhaseOne()`: performs the first commit phase, including journal/database sync work.
- `sqlite-master/src/pager.c:sqlite3PagerCommitPhaseTwo()`: finalizes the commit and returns the pager to a reader/open state.
- `sqlite-master/src/wal.c`: handles WAL write transactions and WAL frame commits when WAL mode is enabled.

### Concrete Source Trace Evidence

`results/source_trace_output.txt` was generated by running the rebuilt custom SQLite binary. The `[SQLITE_TRACE]` lines below are emitted from modified SQLite C source files, so they prove that this run reached the internal transaction path rather than only observing SQLite from the outside.

```text
[SQLITE_TRACE] sqlite3VdbeExec: Starting VDBE execution
[SQLITE_TRACE] OP_Transaction: Starting/Upgrading transaction on DB 0 (P2=0)
```
This proves SQL execution reached SQLite's bytecode engine and opened/upgraded a transaction internally.

```text
[SQLITE_TRACE] pager.c:sqlite3PagerBegin begin state=1 lock=1 exFlag=0 subjInMemory=0 journalMode=0 wal=0
```
This proves the write reached the pager transaction manager in rollback-journal mode.

```text
[SQLITE_TRACE] pager.c:pagerLockDb request state=1 oldLock=1 newLock=2 journalMode=0 wal=0
```
This proves SQLite requested a lock escalation as part of the write transaction.

```text
[SQLITE_TRACE] pager.c:hasHotJournal access rc=0 exists=0 journalOpen=0
```
This proves the rollback-journal path checked whether recovery was needed before continuing.

```text
[SQLITE_TRACE] btree.c:sqlite3BtreeInsert begin root=2 cursorPage=2 key=1 nData=19 flags=0x0 seekResult=-1
```
This proves row insertion reached the B-tree storage path.

```text
[SQLITE_TRACE] pager.c:sqlite3PagerCommitPhaseOne begin state=3 lock=4 journalMode=0 wal=0 noSync=0 dbSize=2 dbOrigSize=2
[SQLITE_TRACE] pager.c:sqlite3PagerCommitPhaseTwo complete rc=0 state=1 lock=1 journalMode=0 wal=0
```
These lines prove that the pager commit protocol reached phase one and phase two successfully.

The main source trace is a rollback-journal trace, so it does not contain WAL frame lines. WAL source-path evidence appears in the WAL vs rollback-journal experiment. This trace does not prove every possible SQLite path; it proves one complete observed transaction path for the workload used in `experiment_execution_trace.py`.

## 3. Design Decisions

### Rollback Journal Atomic Commit

Rollback-journal mode stores enough information to undo uncommitted page changes. If a crash occurs before commit completes, SQLite can use the journal path in `pager.c` to preserve atomicity. The observed DELETE-mode traces show pager begin, lock escalation, commit phase one, and commit phase two.

### WAL Append-Based Commit

WAL mode appends changed page frames to a WAL file instead of immediately overwriting the database file. The WAL experiment shows `wal.c:sqlite3WalBeginWriteTransaction` and `wal.c:walFrames` traces, proving that WAL mode uses a different durability path from rollback journaling.

### Pager Abstraction and Lock Management

The B-tree layer handles logical page/cell changes, but it does not directly own filesystem locking or crash recovery. It asks the pager for writable pages. The pager centralizes page cache state, journal mode, lock acquisition, commit phases, and recovery checks.

## 4. Concept Mapping

| Concept | SQLite implementation | Source reference |
|---|---|---|
| B-tree storage | Tables and indexes are stored as B-tree pages; inserts modify cells in B-tree pages. | `sqlite-master/src/btree.c` |
| Rollback journaling | Old page information is protected so incomplete transactions can be rolled back. | `sqlite-master/src/pager.c` |
| Write-ahead logging | WAL mode appends committed frames to the WAL file. | `sqlite-master/src/wal.c` |
| Fault tolerance / recovery | Hot-journal checks and recovery preserve atomicity after failure. | `pager.c:hasHotJournal()` |
| Locking / concurrency control | Pager lock transitions enforce SQLite's single-writer model. | `pager.c:pagerLockDb()` |
| Execution engine | VDBE opcodes drive transaction execution from parsed SQL. | `sqlite-master/src/vdbe.c` |

## 5. Experiments

The source trace above is execution evidence, not a separate experiment. The experiments below test behavior under different transaction conditions.

### A. Batch-Size Experiment

Script: `experiment_batch_size.py`  
Output: `results/batch_size_output.txt`

Setup from the result file:

- Custom binary: `build/sqlite3_custom`
- Total rows: `2000`
- Journal mode: rollback journal (`DELETE`)
- Synchronous mode: `NORMAL`

Observed results:

| Batch size | Time (s) | Rows/sec | Commits | Trace lines |
|---:|---:|---:|---:|---:|
| 1 | 1.933 | 1,035 | 2000 | 58060 |
| 10 | 0.205 | 9,774 | 200 | 18460 |
| 100 | 0.035 | 56,923 | 20 | 14500 |
| 500 | 0.019 | 104,664 | 4 | 14148 |
| 2000 | 0.015 | 137,879 | 1 | 14082 |

Smaller batches cause more commits. Batch size `1` performed `2000` commits and produced `58060` trace lines. Batch size `2000` performed `1` commit and produced `14082` trace lines. This shows that repeated commits trigger repeated pager, journal, lock, and commit-phase work. Larger batches reduce repeated transaction overhead and improve observed throughput.

Absolute runtime is machine-dependent. The important evidence is the relationship between commit count, throughput, and trace-line count.

### B. Crash-Recovery Experiment

Script: `experiment_crash_recovery.py`  
Output: `results/crash_recovery_output.txt`

Observed result values:

- Setup exit code: `0`
- Writer killed before `COMMIT` after sending `1000` inserts.
- Writer exit code: `-9`
- Writer trace lines: `732`
- Rollback journal present before reopen: `True`
- Verifier exit code: `0`
- Verifier trace lines: `31`
- Final row count after reopen: `0`
- Final result: `PASS: uncommitted rows did not persist after reopening the database.`

Relevant trace evidence:

```text
[SQLITE_TRACE] pager.c:hasHotJournal access rc=0 exists=1 journalOpen=0
```

The writer process was killed before commit, and the rollback journal existed before reopen. After reopening the database, the verifier saw row count `0`. This demonstrates SQLite atomicity for the observed crash case: uncommitted transaction data did not survive.

### C. WAL vs Rollback-Journal Experiment

Script: `experiment_wal_vs_rollback.py`  
Output: `results/wal_vs_rollback_output.txt`

Setup from the result file:

- Rows: `5000`
- Batch size: `200`
- Expected insert transactions: `25`
- SQLite binary: `build/sqlite3_custom`
- Concurrent reader busy timeout: `50ms`

The same write workload was executed twice: once in DELETE rollback-journal mode and once in WAL mode. The experiment combined source-code trace evidence with a concurrent reader that repeatedly issued `SELECT COUNT(*) FROM data` while the writer was active.

Observed comparison:

| Mode | Time (s) | Successful reads | Blocked reads | Trace lines | Pager P1 | Pager P2 | WAL begin | WAL frames |
|---|---:|---:|---:|---:|---:|---:|---:|---:|
| DELETE | 0.077 | 35 | 1 | 35582 | 25 | 25 | 0 | 0 |
| WAL | 0.047 | 47 | 0 | 35472 | 25 | 25 | 25 | 25 |

Both modes still passed through pager commit phase one and phase two exactly `25` times. That is expected because the pager coordinates commit ordering and locking in both rollback-journal and WAL mode; WAL mode additionally enters the WAL write path in `wal.c`.

DELETE mode showed `0` WAL begin traces and `0` WAL frame traces. WAL mode showed `25` WAL begin traces and `25` WAL frame traces, proving that WAL mode entered `wal.c` and used append-based WAL frame writes instead of only relying on the rollback-journal commit path.

Example trace snippets:

```text
[SQLITE_TRACE] pager.c:sqlite3PagerCommitPhaseOne begin ... journalMode=0 wal=0
[SQLITE_TRACE] wal.c:sqlite3WalBeginWriteTransaction begin ...
[SQLITE_TRACE] wal.c:walFrames begin frames=10 isCommit=1 ...
```

Observed behavior also supports the WAL concurrency tradeoff: DELETE mode had `35` successful reads and `1` blocked read, while WAL mode had `47` successful reads and `0` blocked reads. Timings and read counts are machine-dependent, but the trend is consistent with WAL allowing readers to continue more smoothly during writer activity.

### What We Learned

- Small transactions are expensive because each transaction repeatedly enters the pager commit and journaling machinery.
- Rollback journal recovery protects atomicity after failure; in the crash experiment, uncommitted rows did not persist after reopening.
- WAL changes the write path by appending frames and updating the WAL index instead of using only rollback-journal behavior.
- Source instrumentation connected abstract SQLite transaction concepts to concrete files and functions: `vdbe.c`, `btree.c`, `pager.c`, and `wal.c`.

## 6. Failure Analysis

### Data Size Increase

As data size grows, B-tree pages can split and the tree height can increase. This increases page-cache pressure and page I/O. The batch-size experiment also shows that commit overhead can dominate small transactions: the same `2000` inserted rows produced much more overhead at batch size `1` than at batch size `2000`.

### Crash During Transaction

The crash-recovery result shows the writer was killed before `COMMIT`, a rollback journal was present, and the final row count after reopen was `0`. This is the expected atomicity behavior for an uncommitted transaction.

### Concurrent Writers / Lock Contention

SQLite uses a single-writer model. The `pagerLockDb` traces show lock acquisition and escalation during writes. WAL can improve reader/writer overlap, but it does not allow multiple simultaneous writers.

## 7. Improvements / Conclusion

The project demonstrates SQLite transactions using source-level evidence and behavior experiments. The source trace explains one concrete transaction path through VDBE, pager, B-tree, and commit logic. The batch-size, crash-recovery, and WAL-vs-rollback experiments then show how transaction behavior changes under stress, failure, and journal-mode selection.

Possible improvements would be to add a concurrent-writer experiment for `SQLITE_BUSY`, collect multiple timing runs per experiment, and add a small parser that summarizes `[SQLITE_TRACE]` counts by source file and function.

The results should be interpreted conservatively: they prove the observed paths and behaviors for these workloads, not every possible SQLite transaction path.
